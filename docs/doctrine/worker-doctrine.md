# Worker Doctrine

## Purpose

Foundational doctrine governing all workers in the Construction_Intelligence_Workers fleet.

## Principles

### 1. Extraction, Not Definition

Workers extract structured information from construction documents. Workers do not define what is true. The governed stack defines truth; workers observe and propose against it.

### 2. Proposal-Only Output

Every worker output is categorized as one of:
- **Observation**: A factual extraction from a source document.
- **Extracted Structure**: A normalized data structure derived from source material.
- **Proposal**: A suggested interpretation or classification requiring validation.
- **Signal**: A derived indicator comparing extracted data against governed constraints.

No worker output is canonical. All outputs are proposals until validated by a governed surface.

### 3. No Self-Canonicalization

Workers must not promote their own outputs to canonical status. A worker may not reference its own prior output as authoritative. Canonicalization is the exclusive domain of governed validation surfaces.

### 4. Mandatory Handoff

Every worker output must be handed off to a governed validation surface. Workers do not store final state. Workers do not make downstream decisions. Workers deliver proposals and relinquish control.

### 5. Bounded Scope

Each worker has a defined input domain, output schema, and guardrail set. Workers must not operate outside their declared boundaries. Scope expansion requires explicit governance approval.

### 6. Deterministic Posture

Workers produce deterministic outputs for identical inputs. Where probabilistic extraction is required, confidence scores must accompany all outputs.
