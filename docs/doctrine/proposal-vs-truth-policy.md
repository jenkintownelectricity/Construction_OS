# Proposal vs Truth Policy

## Purpose

Defines the boundary between worker outputs (proposals) and governed truth (canonical definitions).

## Policy

### Worker Outputs Are Proposals

All worker outputs are proposals, observations, or signals. They represent extracted or derived information that has not been validated by a governed surface.

A worker output:
- May be correct, partially correct, or incorrect.
- Carries no canonical authority.
- Must not be consumed as truth by any downstream system without validation.
- Must be tagged with its output category (observation, extracted_structure, proposal, signal).

### Truth Is Defined Upstream

Canonical truth originates from governed layers:
- **Layer 0 (Universal_Truth_Kernel)**: Root doctrine and epistemological constraints.
- **Layer 5 (Construction_Kernel)**: Domain kernel definitions for geometry, chemistry, assembly, governance, reality, deliverables, intelligence.
- **Layer 6 (Construction_Runtime)**: Execution contracts, validators, audit surfaces.

Workers may reference upstream truth to inform extraction. Workers may not override, extend, or redefine upstream truth.

### Validation Gate

The transition from proposal to canonical truth occurs exclusively at governed validation surfaces. This gate is not controlled by workers. Workers deliver; validators decide.

### Enforcement

Any worker output that lacks a proposal/observation/signal tag is non-compliant. Any system that consumes unvalidated worker output as canonical truth is non-compliant.
